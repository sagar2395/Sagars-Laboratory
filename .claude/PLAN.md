# Implementation Plan: Sagars-Laboratory

> Last updated: 2026-03-07
> Status: All phases implemented + Operational Testing complete

---

## Vision

A production-quality Platform Engineering lab with:
- **CLI** (`labctl`) for all operations via terminal
- **Web UI** for visual platform overview, scenario management, and cluster visualization
- **Scenario playgrounds** activated with one command, explored interactively
- **Tool swappability** — swap Traefik for Nginx, Prometheus for Victoria Metrics, etc. without rebuilding
- **Multi-runtime** — works on k3d locally, AKS/EKS in the cloud via Terraform

---

## Design Principles

### Tool Swappability
Every platform component is behind an **interface** (abstraction):
```
platform/
  ingress/
    _interface.yaml        # Contract: what an ingress must provide (ports, hostnames, TLS)
    traefik/               # One implementation
    nginx/                 # Another implementation
  monitoring/
    metrics/
      _interface.yaml      # Contract: scrape endpoint, query API, retention
      prometheus/          # Default
      victoria-metrics/    # Alternative
    logging/
      _interface.yaml
      loki/
      elasticsearch/
    tracing/
      _interface.yaml
      tempo/
      jaeger/
```

Active tool selection goes in `.env` or `runtime.env`:
```env
INGRESS_PROVIDER=traefik       # or nginx
METRICS_PROVIDER=prometheus    # or victoria-metrics
LOGGING_PROVIDER=loki          # or elasticsearch
TRACING_PROVIDER=tempo         # or jaeger
GITOPS_PROVIDER=argocd         # or flux
CHAOS_PROVIDER=chaos-mesh      # or litmus
POLICY_PROVIDER=kyverno        # or opa-gatekeeper
SECRETS_PROVIDER=sealed-secrets # or external-secrets
```

`make platform-up` / `labctl platform up` reads these and installs the selected implementation. Swapping is: change the variable, run `labctl platform down && labctl platform up`.

### Production-Ready Quality
- All shell scripts have error handling (`set -euo pipefail`), input validation, colored output
- Helm values are parameterized, not hardcoded
- All tools version-pinned
- CLI has proper error messages, help text, shell completions
- Web UI served from the CLI binary (embedded static assets)
- Health checks and status commands verify every component

## Phase 1: Lab Controller CLI + Web UI

**Goal**: Build `labctl` — a Go binary that provides both CLI commands and a web-based dashboard for visualizing and controlling the lab.

### 1A: CLI Foundation

**Commands** (wrapping existing shell scripts):
```
labctl init / teardown / reset / status
labctl runtime up/down/status
labctl app build/deploy/destroy/list <name>
labctl platform up/down/status
labctl check tools/cluster/ingress
labctl scenario list/info/up/down/status    # wired in Phase 2
labctl ui                                   # start web UI server
```

**Architecture**:
```
cmd/labctl/
  main.go
  cmd/
    root.go, init.go, runtime.go, app.go, platform.go, scenario.go, ui.go
  internal/
    config/         # Reads .env, versions.env, app.env, runtime.env
    executor/       # Wraps os/exec to call existing shell scripts
    runtime/        # Runtime interface (k3d, aks, eks)
    platform/       # Platform component registry + provider resolution
    scenario/       # Scenario engine (Phase 2)
    api/            # HTTP API server for the Web UI
    ui/             # Embedded web UI static assets
```

**Tech choices**:
- Go + cobra (CLI) + viper (config)
- The CLI wraps existing scripts via executor, not replaces them
- Shell completions auto-generated by cobra

### 1B: Web UI

**Tech**: React (or Svelte) SPA, embedded into the Go binary via `embed.FS`

**Served by**: `labctl ui` starts an HTTP server on `localhost:3939` (configurable)

**Pages/Views**:

1. **Dashboard** (home)
   - Cluster status: runtime type, node count, k8s version, context name
   - Platform components: grid of cards showing each component (ingress, metrics, logging, tracing) with health status (green/red/yellow) and which provider is active
   - Deployed apps: list with pod count, status, endpoints, quick actions (scale, restart)
   - Resource usage overview: CPU/memory across nodes (pulled from metrics API)

2. **Scenarios**
   - List of available scenarios with status (active/inactive), category badges
   - Click to see details: description, prerequisites, what gets installed, exploration tips
   - **Start/Stop buttons** to activate/deactivate scenarios
   - Active scenario shows live status of its components

3. **Platform**
   - Component registry: all available providers per category (ingress, monitoring, etc.)
   - Active selection highlighted
   - Swap provider: select alternative, click "Switch" (triggers down+up)
   - Per-component status with logs link

4. **Apps**
   - List of apps from `apps/` directory
   - Deploy/destroy actions
   - Pod status, events, recent logs (last 50 lines via k8s API)

**API layer** (`internal/api/`):
- REST API backing the UI: `GET /api/status`, `GET /api/scenarios`, `POST /api/scenarios/:name/up`, etc.
- Uses the same internal packages as the CLI commands
- WebSocket endpoint for real-time status updates

**Key files**:
- `cmd/labctl/internal/api/server.go` — chi/echo HTTP router
- `cmd/labctl/internal/api/handlers/` — status, scenarios, platform, apps handlers
- `ui/` — frontend source (React/Svelte project)
- `cmd/labctl/internal/ui/embed.go` — `//go:embed` the built frontend

## Phase 2: Scenario Framework + Observability Playground

**Goal**: Build the scenario engine and ship the first playground.

### Scenario Engine

**Location**: `cmd/labctl/internal/scenario/`

**`scenarios/<name>/scenario.yaml`** defines:
```yaml
name: observability-sre
displayName: "Observability & SRE"
description: "Full observability stack with logging, tracing, alerting, SLO dashboards"
category: observability
prerequisites:
  platform: [ingress, monitoring/metrics]
  apps: [go-api]
runtimes: [k3d, aks, eks]
components:
  - name: loki
    type: helm
    chart: grafana/loki-stack
    namespace: monitoring
    valuesFile: values/loki.yaml
  - name: tempo
    type: helm
    chart: grafana/tempo
    namespace: monitoring
    valuesFile: values/tempo.yaml
  - name: alerting-rules
    type: manifest
    path: manifests/alerting-rules.yaml
  - name: slo-dashboards
    type: grafana-dashboard
    path: dashboards/
explore:
  urls:
    - label: "Grafana Dashboards"
      url: "http://grafana.{{.DomainSuffix}}"
  commands:
    - label: "Generate traffic"
      command: "for i in $(seq 1 100); do curl -s http://go-api.{{.DomainSuffix}}/health; done"
  tips:
    - "Open Grafana Explore, select Loki, query {app=\"go-api\"}"
```

### Observability Scenario (`scenarios/observability-sre/`)
- Loki (log aggregation), Tempo (tracing), alerting rules, SLO dashboards
- Go API enhanced with OpenTelemetry tracing + structured JSON logging
- Web UI shows scenario as active with live component health

## Operational Testing Phase (Session #10) — DONE

Real-world testing via the Web UI uncovered systemic issues across the stack:

### Key Lessons Learned

1. **Config propagation is critical**: Go code resolved config correctly but never passed it to shell scripts. Fixed with `exec.SetEnv()` in `root.go`. Rule: ALL config values must flow through the executor environment, not file-based sourcing.

2. **k3d bundles Traefik**: k3d/k3s installs Traefik in kube-system via a HelmChart CRD. Deleting the deployment alone doesn't work — k3s reconciles it back. Must either disable at cluster creation (`--disable=traefik`) or delete the HelmChart CRD first.

3. **Static YAML for ingress doesn't work**: Any ingress YAML file with hardcoded hostnames breaks when domain changes. Use inline heredocs with `$DOMAIN_SUFFIX` interpolation or Helm `--set` overrides.

4. **Success broadcasts are mandatory**: WebSocket-based UIs need explicit success events, not just error events. The scenario status race condition (markActive runs after last component event) required adding post-completion broadcasts.

5. **Archived projects break distribution**: Kubernetes Dashboard moved to `kubernetes-retired`. Both Helm repo (404) and OCI registry (403) stopped working. Direct tarball install from GitHub release is the fallback.

6. **Helm stuck releases**: Interrupted installs leave releases in `pending-install` state, blocking future installs. Pre-check for pending state and clean up before attempting install.

## Tool Swappability Matrix

| Category | Default | Alternatives | Config Variable |
|----------|---------|-------------|-----------------|
| Ingress | Traefik | Nginx, Contour | `INGRESS_PROVIDER` |
| Metrics | Prometheus | Victoria Metrics | `METRICS_PROVIDER` |
| Logging | Loki | Elasticsearch | `LOGGING_PROVIDER` |
| Tracing | Tempo | Jaeger | `TRACING_PROVIDER` |
| GitOps | ArgoCD | Flux | `GITOPS_PROVIDER` |
| Chaos | Chaos Mesh | Litmus | `CHAOS_PROVIDER` |
| Policy | Kyverno | OPA Gatekeeper | `POLICY_PROVIDER` |
| Secrets | Sealed Secrets | External Secrets | `SECRETS_PROVIDER` |
| TLS | cert-manager | — | `TLS_PROVIDER` |

Each provider follows the same contract: `install.sh`, `uninstall.sh`, `status.sh`, `values.yaml`.
